## SHIPPED — F2 leadership section + leader access (deployed + E2E-validated)

- **Gates + helpers (migration `f2_leadership_gates_and_helpers`).** New `bw_is_team_leader_of(p_member)`; new `bw_all_subjects_consent(p_subject_ids, p_audience)` (a subject counts as consenting if `share_ptp_full` OR the audience-specific flag; empty subject set → false; threshold is a **tunable constant**, locked at ALL subjects = 1.0). Extended `bw_can_read_team_profile` (added a team-LEADER branch and a SUPERVISOR branch, each all-members-consent-gated; supervisor branch also requires `supervisor_dashboard_enabled`) and `bw_can_read_paired_profile` (same two branches, **work mode only**). New `bw_can_see_leadership_content(p_profile, p_kind)`: ungated for generator / super-admin / org-coach; consent-gated for org-admin, company-admin, team-leader, supervisor. All branches verified across the full impersonation matrix (leader / supervisor / stranger / member-subject; consent on/off; audience independence; `share_ptp_full` override; supervisor contract dependency) in rolled-back transactions.
- **Server-side enforcement (migration `f2_leadership_section_rls_enforcement`).** Two RESTRICTIVE SELECT policies AND with the existing permissive read gates: `tpsec_leadership_gate` on `team_profile_sections` (`section_type <> 'leadership' OR bw_can_see_leadership_content(team_profile_id,'team')`) and `ppsec_leader_actions_gate` on `paired_profile_sections` (same for `leader_actions` / `'paired'`). For every other `section_type` the predicate is TRUE (no-op); restrictive policies can only REMOVE access. Verified live via role+JWT impersonation: super-admin sees the gated row + normal sections; a plain subject (`corporate_employee`) sees normal sections but NOT the gated row; a stranger sees nothing.
- **Generation.** `generate-team-narrative` v10 → **v11** adds a concise `leadership` section (top 3 things the leader needs to know, each with one move; `{headline, detail, action}` × 3 — the same shape as `team_in_three`, deliberately smaller than the detailed `leader_brief`). `generate-paired-narrative` v14 → **v16** adds a `leader_actions` section (same 3-object shape) for **work** pairs only; all original v14 prompts were preserved verbatim (a v15 that had condensed them was discarded and re-deployed as v16 from an exact v14 base).
- **Frontend (Lovable prompt, applied — commit `ca300d3`, builds clean).** New shared `LeadershipModal` component; entry-point "For the leader" button + modal wired into `TeamReport` (`sections["leadership"]`) and `PairedReport` (`sections["leader_actions"]`, work mode). **Gated on DATA PRESENCE, not the client `canSeePrivileged` flag** — because a team leader is typically a `corporate_employee` (so `canSeePrivileged` is false), and RLS is the authoritative deliverer; gating on the client flag would hide the content from exactly its audience. Export dialog gained "For the leader" toggles (availability-gated via `leadershipAvailable` / `leaderActionsAvailable`); the team + paired PDF generators render the section; the two PDF-data assemblers carry it through without deleting it in the privileged block.
- **Leader entry point — no new code.** `bw_list_my_reports` filters on the two read gates that F2 extended, and it already powers the un-guarded "Team Reports Shared With Me" / "Paired Reports Shared With Me" pages. So a leader/supervisor now automatically sees their team report and their members' work-context paired reports there, with Open links to the report routes (which have no RoleGuard — RLS decides).
- **E2E-validated in the live UI.** Generated a fresh work-context paired report between two test-org members: `leader_actions` generated as a clean array of exactly three `{headline, detail, action}` objects with substantive, leader-specific content. As super-admin (ungated) the "For the leader" button + modal rendered and the Export dialog exposed the toggle.
